Scale Review: What Will Break Before 10,000 Users

JapamApp has a strong early-stage product feel, but the current architecture will become slower, more fragile, harder to secure, and harder to maintain as usage grows. This page is meant to explain those issues clearly, in plain English, so you can see what to fix first.

Audience: junior repo owner Tone: coaching, not blame Target scale: 10,000 users

What This Means In Plain English

The biggest problem is not one bad bug. The real problem is that data loading, sync, storage, and business logic are spread across large screen files and repeated in several places. That creates four kinds of pain at scale: more backend cost, more device work, weaker security boundaries, and slower development when you try to change anything.

Critical

Problem 1: Unbounded Full-History Reads

Several normal app flows fetch or rebuild an entire user's history when they only need a small slice of it.
Main failure: performance

What is happening

Screens like the main tap flow, timer, and history screen repeatedly restore or fetch full history data. That means the amount of work grows with the user's entire lifetime activity, not with what the screen actually needs right now.

Current
Load full history to compute today stats
Better
Fetch one page or one aggregate for today
Junior-dev takeaway: this is like downloading the whole spreadsheet every time you only want today's row.

Why this breaks at 10,000 users

  • screen load cost grows as each user accumulates more history
  • network payloads get bigger over time for routine actions
  • backend reads become more expensive than the UI value they provide

Why This Matters

A slow pattern repeated on every screen eventually feels like the whole app is slow, even if each query “works.”

Recommendation

Add paginated history reads and server-side aggregate endpoints for things like today totals and streaks. Fetch deltas since a watermark instead of reloading everything.

Tradeoff

This adds backend design work, but it gives you predictable performance and lower read cost.

Critical

Problem 2: Client-Enforced Tenancy And Anon-Key Data Access

The app client is still doing too much of the work that the backend should enforce for identity and data isolation.
Main failure: security

What is happening

Many requests send a client-provided user_id together with the anon key. That means the backend is trusting request shape more than it should. For a real product, identity should be derived from authenticated backend context.

App sends
user_id
→
REST call uses anon key
→
DB accepts client identity
Junior-dev takeaway: if the app tells the database who the user is, the app is holding too much security responsibility.

Why this breaks at 10,000 users

  • security rules become easier to bypass accidentally or intentionally
  • shared tables become riskier to operate as real user data grows
  • debugging data access bugs gets much harder because trust boundaries are blurry

Why This Matters

Convenient auth shortcuts are cheap early on, but expensive later because they force the client to be your security layer.

Recommendation

Use authenticated Supabase JWTs or Edge Functions for data access and enforce row-level security from backend user identity, not a client-supplied user_id.

Tradeoff

You will need auth and policy cleanup, but you will have a much safer foundation for real users and real support issues.

Problem 3: AsyncStorage Is Acting Like The Main Database

History and tombstones are stored as growing JSON blobs, then repeatedly rewritten for normal operations.
Main failure: local storage

What is happening

The app often reads the entire history blob, parses it, changes it, stringifies it, and writes the whole thing back. That is okay for tiny prototype data and a poor fit for long-lived user history.

Read whole blob
→
Parse JSON
→
Modify + rewrite
Junior-dev takeaway: when a single row changes, you should not need to rewrite the whole file-sized value.

Why this breaks at 10,000 users

  • save and delete operations become O(n) in total history size
  • older power users get slower behavior over time
  • the app does more device CPU and memory work than necessary

Why This Matters

This is one of the clearest signs that the app has outgrown a prototype storage model.

Recommendation

Move history and tombstones to a row-based local database such as SQLite. Keep AsyncStorage only for small settings and simple flags.

Tradeoff

You will need a migration and a stronger data layer, but row-level updates will stay fast as history grows.

High

Problem 4: Sync Does Too Much Work And The Write Path Is Chatty

The sync model does large reconciliation passes and the app performs more small writes than it should.
Main failure: backend load

What is happening

The app fetches full tombstone or history-related sets, self-heals across large local datasets, and uploads records one-by-one. It also writes timer and totals state often enough to create extra backend traffic.

Current
Many small repeated writes + large sync passes
Better
Delta sync + batch writes at meaningful checkpoints
Junior-dev takeaway: if your sync logic keeps re-checking everything, it is doing too much work for each tiny change.

Why this breaks at 10,000 users

  • backend traffic grows faster than real user value
  • sync logic gets harder to reason about and support
  • mobile data usage and battery cost rise too

Why This Matters

This is not just about cloud cost. Chatty sync also makes correctness harder, because more moving pieces means more ways to drift.

Recommendation

Use delta sync with updated_at and deleted_at cursors, batch uploads/deletes, and save cloud state at semantic checkpoints instead of frequent timer-driven intervals.

Tradeoff

You trade simple brute-force logic for more thoughtful sync design, but the system becomes cheaper and easier to scale.

Problem 5: Core Domain Logic Is Duplicated Across Giant Modules

Important behavior is repeated across a hidden legacy flow, the visible tap flow, and a separate timer engine.
Main failure: maintainability

What is happening

The app has overlapping implementations for auth restore, history restore, session saving, and timer behavior. That means future fixes are more likely to drift or only get applied in one place.

Hidden legacy flow
Visible tap flow
Separate timer engine
Better: one shared history repository + one shared timer domain layer
Junior-dev takeaway: duplicated business rules are a scaling problem too, because they slow the team and create regression risk.

Why this breaks at 10,000 users

  • bug fixes need to be made in multiple places
  • behavior drifts between screens over time
  • feature delivery slows down as the codebase gets more fragile

Why This Matters

Even if infrastructure scaled perfectly, this structure would still slow development and increase production bugs.

Recommendation

Keep one canonical tap flow and extract shared services such as a historyRepository, syncService, and timer controller so screens mostly render state instead of owning business rules.

Tradeoff

This is real refactor work, but it pays back in faster development, fewer regressions, and more confidence when changing behavior.

Fix These First

If you only have time to work on a few things, focus on the foundation first. These steps reduce the biggest long-term risks fastest.

1

Secure backend access and real tenancy

Move data reads and writes behind authenticated backend rules so the app is not acting like the security layer.

2

Stop full-history sync and full-history reads

Add delta sync, pagination, and aggregate reads so the app only loads what the current screen needs.

3

Move history out of raw AsyncStorage blobs

Use a row-based local database for history and tombstones so local work stays cheap as the user keeps using the app.

4

Consolidate the duplicated business logic

After the storage and backend layers are cleaner, unify the repeated logic into shared services so new features do not keep multiplying maintenance cost.

Bottom line: this app does not need a huge rewrite right away. It does need a change in direction. The main shift is from “screen files owning everything” to “shared data and domain layers owning the hard parts.”
