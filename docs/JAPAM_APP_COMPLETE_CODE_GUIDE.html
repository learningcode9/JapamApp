🪔 Japam App — Complete Code Guide

This is a single, offline, self-contained learning file about the Japam (Mantra Japam) React Native / Expo app. You can save this file anywhere and open it by double-clicking — no internet, no login, no Claude subscription required. Everything here was written by reading the actual source code in this repository as it exists today, not from an idealized future design. File paths are repo-relative, e.g. contexts/timer-context.tsx — open that path from the project root to see the real file.

1 App purpose and main features

Japam (జపం) means the repeated chanting of a mantra. Traditionally this is counted using a mala (మాల) — a string of 108 beads, one bead per repetition. This app is a digital assistant for that practice: it counts repetitions for you and keeps a history of your practice over time.

Main features (as they exist in code today)

FeatureWhat it doesMain screen file
Timer JapamPick a duration (e.g. 10 minutes) and chant continuously until the timer completes one "mala" cycle; can auto-repeat for multiple malas.app/(tabs)/timer.tsx
Tap JapamTap a button once per repetition; automatically completes a mala every 108 taps.app/(tabs)/tap-japam.tsx
Manual entry ("Add Japam")Record japam done offline/without the app (e.g. with a physical mala).app/(tabs)/manual.tsx
HistoryView, edit, delete, and export a daily log of completed malas.app/(tabs)/history.tsx
GroupsCreate/join a shared group; members can see each other's practice totals.app/(tabs)/groups.tsx, app/(tabs)/groups-dashboard.tsx
My Japams (workspaces)Multiple named practices (e.g. "Gayatri", "Govinda"), each with its own history/stats.app/my-japams.tsx
SettingsSound/vibration toggles, sign out, account info.app/(tabs)/settings.tsx

The app works fully offline — every action is saved to the phone first, and synced to the cloud (Supabase) only when possible. See §15 Offline-first architecture.

6 Root layout and provider hierarchy

A "Provider" in React lets many screens share the same data without passing it down manually through every component. This app has exactly two of them, both mounted once in app/_layout.tsx:

<TimerProvider>              (contexts/timer-context.tsx)
  <CurrentJapamProvider>     (contexts/current-japam-context.tsx)
    ...the whole app...

TimerProvider — why it exists

The chanting timer must keep working while you switch tabs, lock your phone, or even while Android freezes the JS engine in the background. A timer stored in one screen's local useState cannot survive any of that — so it lives at the very top of the app instead.

Real incident: this provider used to live inside app/(tabs)/_layout.tsx. Expo Router mounted that layout twice during startup, creating two competing TimerProvider instances that both wrote to the same storage keys — causing a running timer to silently reset or change duration. The fix was moving it to the true root layout, which mounts exactly once. See §42.

CurrentJapamProvider — why it exists

Tracks which "workspace" (Japam) is currently selected, so every screen agrees on the same answer. Its own code comment explains it was built this way specifically to avoid a bug class already found once: independent per-screen copies of the same selection going stale relative to each other.

What's notably missing: there is no AuthContext

Login state (userId, userName, guest-or-not) is not a Context anywhere. Every screen independently reads it from AsyncStorage and listens for a plain event named 'japam-auth-updated' (fired via DeviceEventEmitter on native, and a matching window event on web) whenever login state changes. This explains why so many screens repeat the same AsyncStorage.getItem('userId') boilerplate.

8 Components, props, state, hooks, effects, refs, and contexts

Quick refresher, applied to this codebase specifically:

  • Component — a function that returns JSX. Example: export default function TimerScreen() {...} in app/(tabs)/timer.tsx.
  • Props — data passed into a component from its parent. Example: <CurrentJapamHeaderButton variant="timer" /> — variant is a prop.
  • State (useState) — data that changes over time and re-renders the component when it changes. Example: const [seconds, setSeconds] = useState(0) in contexts/timer-context.tsx.
  • Effect (useEffect) — code that runs after render, usually to synchronize with something outside React (storage, timers, event listeners). timer-context.tsx has over 15 separate useEffect blocks — one per concern (AppState listener, notification channel setup, sound loading, cold-start restore, etc.) rather than one giant effect.
  • Ref (useRef) — a mutable value that does not trigger a re-render when changed, and survives across renders. This app uses refs heavily for timer internals (e.g. secondsRef, isRunningRef) specifically so a background event (like a native Android broadcast) can read/update the "real" value without needing a live render cycle.
  • Context (createContext / useContext) — shares data across many components without prop-drilling. This app has exactly two: useTimer() and useCurrentJapam(). See §6.
  • useCallback / useMemo — memoize functions/values so they don't get recreated on every render (mainly used to keep useEffect dependency arrays stable).

9 Timer architecture

What it does

Counts down from a chosen duration (e.g. 10 minutes), plays a completion sound ("Om"), and can automatically repeat for multiple malas, all while surviving app backgrounding and even a full app kill on Android.

Why it exists

A simple in-component countdown breaks the moment you switch tabs or your phone locks. This app needed a timer that is correct even after the JS engine itself is paused or killed by the OS.

Important files

  • contexts/timer-context.tsx — the engine (~2270 lines)
  • lib/timerState.ts — a plain (non-React) object holding the "live" timer numbers, readable outside of React render cycles
  • lib/timerColdStartRestore.ts — pure, unit-tested decision logic for restoring after a real process kill
  • lib/timerForegroundService.android.ts / lib/timerForegroundService.ts — the JS-side bridge to the Kotlin service (real on Android, no-op stub elsewhere)

The three sources of truth for "how much time is left"

1. JS ticker            setInterval, recomputing elapsed = Date.now() - startedAtRef every second.
                        Never trusts an incrementing counter — always anchored to a wall-clock
                        timestamp, so switching apps briefly never desyncs it.

2. Android native       JapamTimerService.kt keeps counting/completing malas even when the JS
   foreground service   engine is fully frozen. Reconciled via reconcileNativeLoops() every time
                        the app returns to foreground: replays any malas Kotlin completed while
                        JS was asleep, then re-anchors the JS ticker to Kotlin's startedAt.

3. AsyncStorage         A snapshot saved every ~10 seconds and on every state change. Read back
   cold-start snapshot  ONLY after a genuine process restart (not a background/foreground cycle).
                        computeColdStartRestoreDecision() decides what to do with it — see §10.

Data flow

User taps Start→timer.tsx calls useTimer().start()→TimerProvider sets refs + state→setInterval ticks every 1s→on completion: saveSession() writes history + syncs

Important functions

FunctionWherePurpose
start / pause / resettimer-context.tsxPublic API exposed via useTimer()
claimCompletionLoop()timer-context.tsx:351-388Single dedup gate — a Set of processed loop numbers — so a mala can never be double-saved no matter which of the 3 time sources triggers completion first
reconcileNativeLoops()timer-context.tsxReplays malas completed natively while JS was asleep, re-anchors the JS ticker
computeColdStartRestoreDecision()lib/timerColdStartRestore.tsPure function deciding whether/how to restore after a real kill

Local storage involved

Keys like timerSeconds, timerRunning, timerTarget, timerPaused, timerCompletedLoops, timerStartedAt, timerSessionId — see §19 for the full table.

Supabase involved

On completion, writes to japam_history (via the same offline-first path as every other feature — see §15) and japam_user_totals (lifetime total cache).

Common failure cases

  • Backgrounding mid-session while a native completion is playing its ~6-second sound — a known past bug, see §42.
  • Two TimerProvider instances mounted at once — a known past bug, now structurally prevented by mounting it only at the true root.
  • Phone call interruptions — cannot be automated/tested reliably (see testing/incidents/known-incidents.json, marked permanently manual-only).

How to test

npm run test:timer runs lib/__tests__/timerColdStartRestore.test.ts and lib/__tests__/malaCompletion.test.ts.

Don't change casually: computeColdStartRestoreDecision() and claimCompletionLoop(). Both exist specifically to prevent silent double-counting or phantom completions — changes here need the accompanying unit tests updated and passing, not just a manual check.

10 Timer start / pause / resume / reset / background / foreground / kill / restore / completion flows

FlowWhat happens
Startstart() in timer-context.tsx sets timerStartedAtRef = Date.now(), begins the JS interval, requests a wake lock, shows a notification, and (Android) calls startForegroundService().
PauseClears the interval, releases the wake lock, hides notifications, and (Android) calls pauseForegroundService(). State is persisted immediately.
ResumeSame as Start, but re-anchors timerStartedAtRef to Date.now() - secondsAlreadyElapsed so the countdown continues from where it left off.
ResetFully clears all timer refs/state and persisted storage; stops the Android service.
App → Background (normal, e.g. switching apps briefly)The JS interval is cleared and the wake lock released (to save battery/comply with OS rules), but the wall-clock anchor and persisted state remain intact. When the app returns to foreground, elapsed time is recomputed from the wall clock — this looks and behaves like the timer kept running in terms of elapsed time, matching the intended rule that ordinary backgrounding must not lose progress.
App → Foreground(Android) reconcileNativeLoops() runs first — replays any malas the Kotlin service completed while JS was asleep. Then, if nothing is running yet, restoreRunningTimerFromStorage() recomputes elapsed time from the wall clock and resumes the JS ticker.
App killed (force-close / OS reclaims memory) then relaunchedThis is the one case that does not auto-resume. computeColdStartRestoreDecision() freezes the timer at the last periodically-saved snapshot (never recomputed from wall-clock dead-time) and shows it as paused, requiring an explicit tap on Resume. This is deliberate — auto-resuming would silently credit/debit the user for however long the app was actually dead.
Mala completioncompleteCycle()/claimCompletionLoop() fire feedback (sound/haptics) and saveSession() (writes history), then either starts the next mala or fully stops if this was the final one.
The one rule that matters most: normal backgrounding (switching apps, screen lock) must never lose progress; only a genuine kill/interruption freezes it, and even then it must never silently auto-resume or auto-complete.

11 Android foreground service

What it does

A "foreground service" is Android's way of letting an app keep running a task with a visible, persistent notification even when the app itself is in the background. This app uses one to keep the Timer running and completing malas even if the JavaScript engine is fully frozen (which Android does aggressively to save battery).

Important files

  • android-native/JapamTimerService.kt (~523 lines) — the actual Kotlin service
  • android-native/JapamTimerModule.kt (~169 lines) — exposes JS-callable methods (the "bridge")
  • android-native/JapamTimerPackage.kt — registers the module with React Native
  • plugins/withJapamTimerService.js — an Expo config plugin that copies these files into the native Android project during expo prebuild, and declares the service + permissions in AndroidManifest.xml
  • lib/timerForegroundService.android.ts — the JS side of the bridge

How it works

  • Started via startForegroundService()/startForeground() with type FOREGROUND_SERVICE_TYPE_MEDIA_PLAYBACK (required on Android 14+, or the OS kills the service with a SecurityException).
  • A repeating tickRunnable checks every second using wall-clock math (durationMs - (now - startedAt)) — not a JS timer.
  • Shows a notification with the current "Mala X / Y" and a live countdown, with Pause/Resume/Stop buttons.
  • On mala completion (handleLoopComplete): if the app is currently in the foreground (checked via a SharedPreferences flag), it deliberately does nothing and lets JS handle the completion — avoiding double-counting. Only if the app is backgrounded does it increment the loop count itself, play the sound, vibrate, and post a completion notification.
  • Persists its own state into Android SharedPreferences (a Kotlin-side key-value store, separate from the app's AsyncStorage) so JS can read it back via getState().

JS ↔ Kotlin bridge methods

Events flow back to JS via a single event: DeviceEventEmitter listens for 'japamTimerLoopComplete', emitted by the Kotlin module when a mala completes while the app was backgrounded.

Common failure cases

The completion sound can take up to ~6 seconds to finish playing on the native side; a past bug treated "sound still playing" as "timer still running," leaving a stale in-progress display after all loops were actually done. See §42.

Don't change casually: this file must stay in sync with AndroidManifest.xml permissions (FOREGROUND_SERVICE_MEDIA_PLAYBACK) declared in app.json. Removing/renaming a bridge method without updating both the Kotlin and TypeScript sides will crash silently on Android only (won't show up in web/iOS testing).

12 Tap Japam architecture

What it does

User taps a big circle once per repetition; the app counts up and automatically completes a "mala" every 108 taps.

Why it exists

Some practitioners prefer counting by touch rather than a timer — this mirrors using a physical mala bead-by-bead.

Important files

app/(tabs)/tap-japam.tsx (~3570 lines, component internally named JapamMain) and lib/malaCompletion.ts (the pure crossing-detection logic it shares with other screens).

Data flow (button tap → database → screen refresh)

User taps the circle
   │
   ▼ handleTap()  (tap-japam.tsx)
   Optimistic local counter update — the number on screen changes INSTANTLY, no waiting
   │
   ▼ detectMalaCrossing(previousTotal, nextTotal)   (lib/malaCompletion.ts)
   A "crossing" check (floor-division), NOT an exact-multiple check — this matters: it catches
   a mala boundary even if a duplicate/missed tap event jumps the total past 108 without landing
   exactly on it.
   │
   ├─ no boundary crossed → just a small tap animation/haptic, done
   │
   └─ boundary crossed → runMalaCompletion({ guard, save, playFeedback })
         The "guard" is checked+marked SYNCHRONOUSLY before anything async starts, so a duplicate/
         concurrent call for the same boundary is rejected immediately — no race condition.
         │
         ├─ save(): appendCompletion() → AsyncStorage['history'] write (the local "truth")
         │     → fires events so History/Stats screens refresh WITHOUT any network round trip
         │
         └─ playFeedback(): Om sound + haptics, runs in parallel — a failure here never blocks
               the save, and vice versa (Promise.allSettled)
   │
   ▼ (separate, "fire and forget" — not awaited by the UI)
   POST to Supabase japam_history — only if a login session exists; otherwise the record just
   stays "pending" and uploads later. See §15.

Common failure cases

Rapid/duplicate taps landing exactly on or near a 108 boundary — this is precisely what the crossing-based (not exact-equality) detection was built to survive. See lib/__tests__/malaCompletion.test.ts's "rapid taps cannot skip completion" test.

17 Completion IDs and duplicate prevention

What it does

Every history record gets a completionId — a stable, deterministic string, not a random one. It is the single most important idea protecting this app from double-counting.

How it's built

  • makeCompletionId(userId, dateISO) — for manual/history-screen entries.
  • makeLoopCompletionId(userId, sessionId, loopNumber) — for Timer completions, one per mala within a session.

18 Tombstones and deletion sync

What it does

A "tombstone" is a record that says "this completion was deleted" — kept separately from the history data itself, so deletion can sync across devices without the deleted item ever quietly reappearing.

Why it exists

If deletion just meant "not in my local list," a device that was offline during the delete would re-download the "deleted" row from Supabase and resurrect it. A tombstone is checked and applied on every merge, so a deleted record stays deleted everywhere.

How it works

1. Local delete:
   AsyncStorage['deletedCompletions'] ← add completionId   (written BEFORE removing the row itself,
                                                              so a same-tick remote refresh can't
                                                              resurrect it)
   AsyncStorage['history']            ← row removed

2. Remote delete (single atomic call):
   supabase.rpc('delete_history_completions', { p_completion_ids })
      → one Postgres transaction that BOTH inserts into deleted_completions
        AND deletes the matching japam_history row(s)

3. Every future merge:
   mergeTombstones(local, remote)  → union of both tombstone lists
   applyTombstones(records, set)   → filters OUT any record whose id is tombstoned
Past bug this fixed: before this RPC existed, the tombstone-write and row-delete were two separate network calls. An interruption between them left a "zombie" row — tombstoned but still physically present in the database, which could inflate anything that directly sums japam_history (like the Groups dashboard). See db/atomic_delete_history_rpc.sql and §42.
Still-open related bug: per testing/incidents/known-incidents.json's group-dashboard-tombstone-mismatch entry — the get_group_dashboard RPC does not currently filter out tombstoned rows from its totals, despite a migration file claiming it does. This is a documented, currently open discrepancy, not something I'm speculating about.

23 Session JWT versus anon key

Supabase gives every request one of two kinds of credential:

CredentialWhat it provesUsed for
Anon key (EXPO_PUBLIC_SUPABASE_ANON_KEY)"This request comes from our app" — nothing about who the user isThe base apikey header on every request; by itself only allowed to do whatever the anon database role's RLS policies permit
Session JWT (Authorization: Bearer <token>)"This request comes from user X, who is currently signed in" (or anonymously authenticated)Required for any write that must be scoped to "my own rows" — the database checks auth.uid() against this token

Throughout the sync code (history uploads, tombstone pushes, manual entry sync), you'll see a consistent pattern: const sessionToken = (await supabase.auth.getSession()).data.session?.access_token; if (!sessionToken) return; — the request is skipped and left "pending" rather than falling back to the anon key, because the anon key has no per-user write permission once RLS is properly scoped.

Currently open, documented security finding: testing/incidents/known-incidents.json's history-rls-anon-open-access entry states that on the current production policies, the anon role can in fact SELECT/INSERT any user's japam_history rows — policy names literally contain the word "test," suggesting temporary rules that were never removed. 26 RLS policies and why they matter

RLS (Row Level Security) is Postgres's built-in permission system: even if a request reaches the database, RLS decides which rows it's allowed to see or change, based on who's asking. Without a matching policy, RLS denies by default.

TableCurrent policy shape (as read directly from schema.sql)
japam_history (production)Loose — SELECT/INSERT open to both authenticated AND anon with no per-user filter at all. Only UPDATE/DELETE are scoped to auth.uid().
japam_history (schema_staging.sql)Tighter — SELECT/INSERT restricted to authenticated only, scoped to auth.uid() matching user_id.
deleted_completions, japam_timer_state, japam_user_totals, user_profilesPermissive "test" policies — broad anon access gated on nothing (USING (true)). Policy names literally contain the word "test."
groups, backup tablesRLS enabled but no policies at all defined — meaning ordinary anon/authenticated roles cannot query them directly; all access must go through the SECURITY DEFINER RPCs (§27).
Why this matters, concretely: the app's public "anon key" is bundled inside the app binary — anyone can extract it. If a table's RLS allows anon to read/write without a per-user filter, anyone with that key (not just this app) can read or write any user's rows on that table directly, bypassing the app UI entirely. This is exactly what testing/incidents/known-incidents.json documents as a currently open, confirmed, critical finding for japam_history in production. See §44.

29 Provider/manual display-name resolution

What it does

Decides what name to show for a person — e.g. on the Groups dashboard — when they haven't explicitly set one.

A real, documented postmortem

Lesson recorded in the doc itself: prefer a server-side, deterministic read over a client-maintained cache whenever the server already owns the underlying data — this avoids an entire class of "was the sync guaranteed to have run" bugs.

31 Current Japam selection

CurrentJapamProvider (contexts/current-japam-context.tsx) is the single global source of truth for "which workspace am I looking at right now," exposed via useCurrentJapam(). 39 How to investigate a sync bug

  1. Determine which side is wrong: is the record missing locally (never saved), stuck pending (never uploaded), or duplicated (uploaded twice)?
  2. For "stuck pending": check whether a valid session exists (supabase.auth.getSession()) at the time of sync — a missing/expired session is a known, currently silent failure mode (§16, §23).
  3. For "duplicated": check whether the record's completionId is actually stable across the two copies — if they differ, something bypassed the deterministic id generation (§17).
  4. For "record vanished after sync": check the tombstone table (§18) — a record can be legitimately removed by a tombstone from another device.
  5. Remember: the sync-upload logic is duplicated across ~6 files (§15, §43) — a bug fixed in one file's copy may still be present in the others.

42 Important production incidents and lessons learned

All entries below are read directly from this repo's own incident documentation (docs/BUGFIX_*.md, docs/investigations/*, testing/incidents/known-incidents.json) — not speculation.

Duplicate TimerProvider mounts (Issues #19/#20)

Expo Router mounted the tab layout twice at startup; two competing TimerProvider instances both wrote to the same storage keys, causing a running timer to appear to reset or change duration. Fix: moved the provider to the true root layout (mounts exactly once). Lesson: global providers belong at the true app root, never in a layout that route resolution could duplicate; lifecycle bugs need real-device validation, not just logs.

Duplicate completion records from process-restart races

Three duplicate-save guards existed only in memory, resetting on every JS restart — a restart at the wrong moment could re-save the same mala with a new, wall-clock-based id. Fix: deterministic completionIds (§17). Known limitation: does not retroactively repair already-affected devices — remediation for existing affected users was designed but explicitly shelved (insufficient evidence, unwillingness to risk hiding legitimate history on unproven signals).

Timer restore showing a stale, already-completed session

A race between the Android completion sound (~6s to finish) and JS reconciliation left stale "in progress" state visible after all malas were actually done. Fix: simplified the cleanup condition to not depend on the native "still running" flag.

Groups dashboard showing an incorrect display name

See §29 for the full story — moved from a write-once client snapshot to a server-side, deterministic name-resolution RPC.

Groups crash / blank screen reports — unresolved

Reported on specific "Family"/"Test" groups; never reproduced on a test device. Explicit conclusion in the doc: "treat as unreproduced, not fixed." An error boundary was added as containment only.

Timer background-pause reports — unresolved

Structured lifecycle testing (lock/unlock, force-stop, swipe-away) passed all scenarios on a test device; explicit conclusion: "No timer code change is justified from this evidence alone."

Offline sync 401 on reconnect

The regular history-upload loop was found to hardcode the anon key instead of the session token in one code path, causing pending uploads to fail with 401 indefinitely. Later addressed via a guarded RLS/JWT migration (git commits 3189861, 8f4e95d, 672c0e6).

OTA adoption crash

An audio/player teardown running on the wrong thread during OTA update-adoption could crash the app outright. Fixed in commit 7fe858f; device-only, no permanent automated test exists (by design).

Recurring lesson across almost all of these: small, focused fixes; real-device validation for anything lifecycle-related; do not guess a fix without a reproduction; and prefer a single server-side source of truth over client-side synchronization whenever the server already owns the data.

46 Frequently asked questions

Why does the timer sometimes show "Resume" instead of "Start" after I force-close the app?

This is intentional. A genuine app kill freezes the timer at its last saved snapshot and requires an explicit Resume tap — it deliberately never auto-resumes counting time that passed while the app was dead. See §10.

Why are there two almost-identical screens for tap counting?

app/(tabs)/tap-japam.tsx is the real, visible tab. app/(tabs)/index.tsx is a hidden legacy duplicate, still reachable via one push call. See §5, §12, §43.

Why does a manual entry always say "Saved" even if I'm offline?

Because the local save (instant, on-device) is what "Saved" refers to — the Supabase upload is a separate, best-effort background step that retries later. See §13, §15.

Can I see my Group's stats scoped to just one of my Japams?

No — Groups sum a member's entire history across all their personal workspaces; there's no per-workspace Group view today. See §28.

Is my data safe if I delete a history entry offline?

Yes — a tombstone is written locally before the entry is removed, and it's pushed to Supabase (atomically, along with the row delete) the next time you're online. See §18.

Why does the app need my Google account's SHA-1 fingerprint?

Google requires this for native Android sign-in to trust the app; without registering it, sign-in fails with "app not authorized." See §22.
